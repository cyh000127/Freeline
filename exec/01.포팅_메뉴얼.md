# Freeline 포팅 메뉴얼

## 1. 문서 개요

본 문서는 GitLab에서 소스를 클론한 뒤 Freeline 서비스를 직접 빌드하고 배포할 수 있도록 정리한 포팅 가이드다.  
대상 시스템은 다음 4개 애플리케이션으로 구성된다.

- Backend: Spring Boot API 서버
- User App: Expo 기반 모바일/웹 앱
- Booth App: 부스 관리자용 Next.js 웹
- Super App: 행사 관리자용 Next.js 웹

운영 기준으로는 별도 인프라(PostgreSQL, Redis, RabbitMQ, Hadoop/Hive, Jenkins, Infisical, Nginx)를 함께 사용한다.

## 3. 서버 구성 요약

기본 운영 구조는 2대 서버 분리형이다.

- Application Server
  - Nginx
  - Backend
  - Frontend(User/Booth/Super)
  - PostgreSQL
  - Redis
  - RabbitMQ
  - Flume Agent
- Infra/Analytics Server
  - Jenkins
  - Infisical
  - Prometheus / Grafana
  - Hadoop(HDFS, YARN, Hive)
  - Loadtest

단일 서버 포팅도 가능하지만, 그 경우 `infra/nginx/upstream.env`, `infra/hosts.conf`, Hadoop/Hive 접속 URL을 새 환경에 맞게 수정해야 한다.

## 5. 빌드 및 배포 절차

### 5.1 공용 Docker 네트워크 생성

모든 compose 서비스는 외부 네트워크 `infra-public`을 사용한다.

```bash
cd infra/networks
docker compose up -d
```

### 5.2 백엔드 의존 인프라 실행

운영 기준 시크릿은 Infisical `/be` 경로에서 주입한다.

```bash
cd infra/backend
infisical run --path="/be" -- docker compose up -d freeline-db freeline-redis freeline-rabbitmq
```

Infisical을 사용하지 않을 경우, 같은 환경변수를 셸 export 또는 `.env` 파일로 주입한 뒤 `docker compose up -d`를 실행하면 된다.

### 5.3 AI Vision 서버 빌드 및 실행

```bash
docker build -t freeline-ai-vision:latest ai/vision-server
cd infra/ai-vision
docker compose up -d
```

### 5.7.1 Firebase FCM 시크릿 등록 방식

이 프로젝트의 백엔드는 Firebase 서비스 계정 JSON 파일을 서버 경로에서 직접 읽지 않는다.  
운영/포팅 시에는 JSON 원본 내용을 Base64 문자열로 변환한 뒤, 시크릿 저장소(기존 운영은 Infisical `/be`)의 `FIREBASE_SERVICE_ACCOUNT_BASE64`에 저장해서 사용한다.

등록 절차 예시:

```bash
# Linux / macOS
base64 -w 0 service-account.json
```

```powershell
# Windows PowerShell
[Convert]::ToBase64String([IO.File]::ReadAllBytes("service-account.json"))
```

위 결과 문자열을 아래 두 값으로 등록한다.

- `FIREBASE_PROJECT_ID`
- `FIREBASE_SERVICE_ACCOUNT_BASE64`

애플리케이션 시작 시 Spring은 이 값을 읽어서:

1. 공백/개행 제거
2. Base64 디코딩
3. 메모리 상의 `ByteArrayInputStream`으로 변환
4. `GoogleCredentials.fromStream(...)`으로 Firebase 초기화

즉, 백엔드 서버에는 Firebase JSON 파일을 별도 경로에 배치할 필요가 없다.

### 5.8 리포트 기능용 Hadoop/Hive 초기화

리포트 기능을 쓸 경우 최초 1회 아래 작업이 필요하다.

```bash

docker exec hive-server beeline -u jdbc:hive2://localhost:10000 -f /scripts/init-hive.sql
```

## 7. 배포 시 특이사항

### 7.1 스키마 관리 방식

- DB 마이그레이션 툴(Flyway, Liquibase)을 사용하지 않는다.
- 기준 스키마는 `be/Freeline/src/main/resources/ddl.sql` 단일 파일이다.
- 애플리케이션 시작 시 `spring.sql.init.mode: always`로 스키마를 적용한다.
- JPA 기본 설정은 `ddl-auto: none`이다.

### 7.2 포트 및 네트워크

- PostgreSQL: `127.0.0.1:5432`
- Redis: `127.0.0.1:6379`
- RabbitMQ: `127.0.0.1:5672`, 관리 UI `127.0.0.1:15672`
- Backend: `8080`
- AI Vision: `8000`
- Nginx: `80`, `443`

DB/Redis/RabbitMQ는 운영 compose에서 loopback에 바인딩되어 외부 직접 노출을 막는다.

### 7.3 프론트엔드 basePath

- Booth 앱: `/booth`
- Super 앱: `/super`

Nginx 또는 별도 리버스 프록시를 사용할 때 반드시 이 basePath를 유지해야 한다.

### 7.4 기존 운영 도메인 하드코딩 주의

- Booth/Super 앱은 `NEXT_PUBLIC_API_URL`이 없으면 저장소에 포함된 기존 운영 API 주소를 기본값으로 사용한다.
- Mobile 앱은 `API_URL`이 없으면 저장소에 포함된 기존 운영 API 주소를 기본값으로 사용한다.
- 따라서 새 서버나 새 도메인으로 포팅할 때는 빌드 전에 반드시 `NEXT_PUBLIC_API_URL`, `API_URL`을 새 주소로 주입해야 한다.

### 7.5 운영 배포 순서

권장 순서:

1. `infra/networks`
2. `infra/backend`
3. `infra/ai-vision`
4. `infra/frontend`
5. `infra/nginx`
6. `infra/hadoop`, `infra/flume-agent`, `infra/monitoring` (리포트/관제 필요 시)

### 7.6 헬스 체크 포인트

- Backend: `/actuator/health`, `/swagger-ui.html`
- AI Vision: `/health`
- Frontend: Nginx 경유 `/`, `/booth`, `/super`
- Hadoop: NameNode UI `:9870`, HiveServer2 `:10000`

### 7.7 모바일 앱 배포 특이사항

- Android FCM 연동용 `google-services.json`이 필요하다.
- `google-services.json`은 모바일 빌드용 파일이고, 백엔드의 FCM 발송용 시크릿은 별도로 `FIREBASE_SERVICE_ACCOUNT_BASE64`를 사용한다.
- 저장소 기본 설정에는 기존 운영 API 주소가 남아 있을 수 있으므로, 포팅 시 `API_URL`을 새 환경 주소로 반드시 재지정해야 한다.
- 로컬 백엔드 연결 시 `API_URL` 또는 `src/constants/env.ts` 기준 값을 맞춰야 한다.

## 8. Jenkins 기준 배포 방식

원 저장소의 CI/CD는 폴더 단위 파이프라인으로 분리되어 있다.

- `be/Freeline/Jenkinsfile`: Backend CI/CD
- `fe/Jenkinsfile`: Frontend CI/CD
- `ai/Jenkinsfile`: AI Vision CI/CD
- `infra/Jenkinsfile`: Infra CI/CD

특징:

- `main` 브랜치만 실제 배포
- Merge Request에서는 테스트/빌드 중심
- `fe/types/` 변경 시 모든 프런트엔드 앱 재빌드
- 인프라 배포는 `infra/hosts.conf` 기준 원격 SSH 배포

## 9. DB 덤프 파일 복원

제출 산출물에 포함한 최신 덤프 파일은 `exec/freeline_db_dump_2026-03-30.dump`다.

복원 예시:

```bash
docker cp exec/freeline_db_dump_2026-03-30.dump freeline-db:/tmp/
docker exec freeline-db sh -lc '
  dropdb -U "$POSTGRES_USER" --if-exists "$POSTGRES_DB" &&
  createdb -U "$POSTGRES_USER" "$POSTGRES_DB" &&
  pg_restore -U "$POSTGRES_USER" -d "$POSTGRES_DB" --clean --if-exists /tmp/freeline_db_dump_2026-03-30.dump
'
```

주의:

- 복원 전 Backend 컨테이너는 중지하는 것을 권장한다.
- 분석 리포트용 HDFS DB dump(`event_*.tsv`)와 PostgreSQL 전체 dump는 별개다.
